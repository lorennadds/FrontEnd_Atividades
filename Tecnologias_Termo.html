TECNOLOGIAS DO TERMO

como o jogo foi construído, peça por peça

Este jogo é um clone do Termo (a versão brasileira do Wordle), construído inteiramente com React. A ideia aqui é abrir o capô e explicar por que cada tecnologia foi escolhida e qual problema ela resolve dentro do jogo.

React

O jogo inteiro vive dentro de um único componente funcional, Termo. Isso faz sentido porque o Termo é, na essência, uma máquina de estados visual: a cada tecla pressionada, só uma parte da tela muda (a linha atual, o teclado, uma mensagem), e o resto continua igual. É exatamente o tipo de problema que o React resolve bem — você descreve como a tela deve ficar para um determinado estado, e o React cuida de atualizar só o necessário no DOM.

Hooks utilizados

useState

Cada pedaço de informação que muda durante a partida vira um estado independente:

  • alvo a palavra secreta sorteada no início da partida
  • tentativas lista das linhas já enviadas, com seu resultado
  • atual a palavra sendo digitada na linha ativa
  • status "jogando", "venceu" ou "perdeu"
  • estadoTeclas a cor que cada tecla do teclado virtual deve assumir
  • mensagem aviso temporário, tipo "Faltam letras"

Separar em vários estados pequenos, em vez de um objeto único, deixa cada atualização mais simples de entender e evita re-renderizações desnecessárias.

useEffect

Usado para registrar um listener de teclado físico (window.addEventListener('keydown', ...)) assim que o componente entra na tela, e removê-lo quando sai. Sem isso, o jogo só responderia a cliques no teclado virtual, ignorando o teclado de verdade.

useCallback

As funções enviarTentativa e handleTecla são memorizadas com useCallback para não serem recriadas a cada renderização — importante porque handleTecla é usada dentro do useEffect do teclado físico. Sem essa memorização, o listener teria que ser removido e recriado a cada tecla digitada.

Lógica do jogo

A função avaliarTentativa é o coração do Termo. Ela compara, letra por letra, a tentativa do jogador com a palavra-alvo, em duas passadas:

  1. Marca como correta toda letra que está na posição certa.
  2. Para as letras que sobraram, verifica se existem em outro lugar da palavra-alvo (e ainda não foram usadas), marcando como presente.
correta presente ausente
Esse cuidado em duas passadas é necessário pra lidar direito com palavras de letras repetidas — um erro comum em implementações apressadas do Wordle, que costumam marcar letras repetidas como presentes mais vezes do que deveriam.

Estilização

Não foi usada nenhuma biblioteca de CSS. Os estilos vivem num arquivo Termo.css separado, aplicado via classes (className) no JSX — sem inline styles espalhados pelo componente. Isso deixa a lógica do jogo e a aparência dele em arquivos diferentes, facilitando trocar uma sem mexer na outra.

A identidade visual escolhida — fundo bege claro, título serifado e letras do tabuleiro em fonte monoespaçada — remete à estética de jornal impresso, reforçando a ideia de "jogo de palavras".

Por que não usar Redux ou Zustand?

Porque não é necessário. Todo o estado do jogo é local a um único componente — não há dados compartilhados entre telas distantes nem necessidade de persistência complexa. Trazer uma biblioteca externa de gerenciamento de estado aqui seria complexidade desnecessária para o tamanho real do problema.
